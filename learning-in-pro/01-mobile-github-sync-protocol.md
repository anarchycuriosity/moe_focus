# 01 - 移动端 GitHub 同步协议：为什么不是“直接复制数据库”

## 前置知识

学习这一节前，你需要先知道几个很基础但重要的概念：

1. 数据库表：像 Excel 表格一样保存数据，每一行是一条记录。
2. 主键：一行数据的身份标识，例如 `id = 3`。
3. UUID：全局唯一标识，适合跨设备识别同一条记录。
4. API：程序之间通过网络约定好的调用方式。
5. 幂等：同一个操作执行多次，结果仍然稳定，不会越做越错。

## 阶段一：打破抽象 —— 探寻设计者动机

### 1. 这个机制解决什么工程痛点

桌面端和手机端都会产生专注记录。如果只把手机数据库复制到桌面端，桌面端原来的数据可能被覆盖；如果只把桌面数据库复制到手机端，手机刚记录的专注也可能丢失。

所以我们真正需要的不是“复制”，而是“合并”。

### 2. 如果设计得很简陋会怎样

假设桌面端有：

```text
id = 1, subject = C++, actual_duration_sec = 1500
```

手机端也有：

```text
id = 1, subject = 网络, actual_duration_sec = 1800
```

如果我们用自增 ID 判断“它们都是 id=1，所以是同一条记录”，那其中一条就会被覆盖。反过来，如果每次同步都简单追加，重复同步三次后，一条 25 分钟记录可能变成 75 分钟。

这就是同步协议最常见的灾难：丢数据或重复数据。

### 3. 在系统底层大致对应什么

在这个项目里，同步不是 Linux 内核系统调用层的问题，而是应用层协议问题：

1. SQLite 负责本地持久化。
2. GitHub Contents API 负责远程读写文件。
3. UUID 负责跨设备身份识别。
4. JSON 文件负责表达可合并的数据集合。

## 阶段二：理论落地 —— 最小可行性项目拆解

### MVP 项目

写一个 200 行以内的 C++ 或 TypeScript 小项目，模拟两个设备同步学习记录：

```text
device_a.json
device_b.json
remote.json
```

每条记录有：

```json
{
  "uuid": "abc",
  "subject": "C++",
  "minutes": 25
}
```

同步规则：

1. 读取本地 JSON。
2. 读取远程 JSON。
3. 用 UUID 做键合并。
4. 本地覆盖同 UUID 的远程版本。
5. 写回远程 JSON。

### 模块划分

| 模块 | 职责 |
| --- | --- |
| `record_store` | 读取和写入 JSON |
| `sync_engine` | 按 UUID 合并 |
| `main` | 构造测试数据并打印同步结果 |

### 容易踩坑的边界

1. 没有 UUID：只能靠标题或时间猜测身份，会误合并。
2. 重复追加：同一条记录每同步一次就多一份。
3. 冲突策略不明确：两台设备同时修改同一条记录时，谁覆盖谁必须有规则。
4. Token 泄露：GitHub Token 不能写进仓库。

## 阶段三：实践作业

### 基础作业

实现一个最小同步程序：

1. 构造两个 JSON 文件。
2. 每个文件有 3 条记录，其中 1 条 UUID 相同。
3. 合并后打印最终记录数。
4. 重复运行同步 3 次，验证记录数不会增加。

### 进阶作业

给每条记录增加 `updated_at` 字段，实现规则：

```text
同 UUID 时，updated_at 更新的一方获胜。
```
