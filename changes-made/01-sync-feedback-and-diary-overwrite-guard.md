# 01 - 同步反馈与日记覆盖保护：让用户知道系统正在做什么

## 一、问题背景

这次修复两个用户体验问题：

1. 点击「同步数据」后，用户很难判断同步是否成功、失败，或仍在进行中。
2. 点击「生成今日」会直接生成日记。如果当天已经存在日记，生成过程可能覆盖用户手写过的内容。

这两个问题看起来像界面小瑕疵，但本质上都属于一个更基础的软件工程原则：**任何耗时操作和破坏性操作，都必须给用户明确反馈**。

## 二、前置知识

### 2.1 状态不是文字，而是有限状态机

对初学者来说，容易把提示语理解成“显示一句话”。但在真实应用中，提示语背后其实是一个有限状态机：

```text
idle -> loading -> success
                -> error
```

同步按钮尤其典型：

- `idle`：用户尚未点击。
- `loading`：同步正在执行，按钮应禁用，防止重复点击。
- `success`：同步结束，告诉用户导入了什么。
- `error`：同步失败，告诉用户为什么失败，至少给出排查方向。

如果只写 `console.log()`，开发者能看到，用户看不到。用户看不到的成功，等价于没有成功反馈。

### 2.2 破坏性操作必须二次确认

生成日记不是简单的“新增”。如果当天已经有日记，重新生成就可能覆盖旧文件。这类操作叫做破坏性操作。

常见破坏性操作包括：

- 删除数据。
- 覆盖文件。
- 重置配置。
- 同步时用远端版本覆盖本地版本。

这些操作不应该直接执行，而应该先让用户知道后果，并给用户一次取消机会。

### 2.3 派生数据也可能包含人工编辑

日记文件看似由程序生成，是“派生数据”。但用户可能打开 Typora 后手写内容，把它变成“程序生成 + 人工编辑”的混合文件。

这就是危险点：

```text
focus_sessions  ->  DiaryService.generate()  ->  sums/YYYY-MM-DD.md
                                              ->  用户手动编辑
```

程序如果再次生成同一天日记，就会把用户后来写的内容覆盖掉。

## 三、这次的解决思路

### 3.1 同步按钮反馈

修复前：

- 点击同步后只有很弱的文字提示。
- 成功和失败的视觉差异不明显。
- 提示会自动消失，用户可能错过。

修复后：

- 点击后立即显示“正在同步数据，请稍等...”。
- 同步成功显示“同步成功: ...”，包含导入会话、同步日记、新文件、清理记录等细节。
- 同步失败显示“同步失败: ...”，保留错误信息。
- 完成后的提示不会立刻消失，用户可以手动关闭。

### 3.2 日记生成保护

修复前：

```text
点击生成今日 -> 直接调用 diary.generate(date) -> 覆盖文件
```

修复后：

```text
点击生成今日
  -> 打开确认窗口
  -> 如果当天已有日记，明确提示会覆盖
  -> 用户确认后才生成
  -> 生成成功后打开 Typora 供用户审核
```

这里用了两层保护：

1. **事前确认**：告诉用户可能覆盖已有日记。
2. **事后审核**：生成完成后打开 Typora，让用户立即检查内容。

## 四、关键代码结构

同步反馈使用一个对象保存提示状态：

```ts
type SyncNoticeType = 'info' | 'success' | 'error'

interface SyncNotice
{
  type: SyncNoticeType
  text: string
}
```

日记生成也使用类似结构：

```ts
type GenerateNoticeType = 'success' | 'error'

interface GenerateNotice
{
  type: GenerateNoticeType
  text: string
}
```

这比单独保存一个字符串更可靠，因为 UI 可以根据 `type` 决定颜色、边框和语义。

## 五、为什么这不是小题大做

对用户而言，软件最可怕的不是失败，而是**不知道发生了什么**。

同步失败但界面没有反馈，用户会以为数据已经同步，从而在另一台设备上继续工作，最后造成数据不一致。

日记覆盖没有确认，用户可能失去当天手写内容。即使技术上可以从 Git 历史恢复，对普通用户来说也已经是一次灾难体验。

因此，好的界面不是装饰，而是数据安全的一部分。

## 七、项目作业

做一个简化版“笔记生成器”：

1. 用户可以输入日期和笔记内容。
2. 点击“自动生成”时，如果当天已有笔记，必须弹出确认窗口。
3. 生成成功后显示绿色状态条。
4. 生成失败后显示红色状态条。
5. 状态条必须保留在页面上，用户可以手动关闭。

进阶要求：

1. 在覆盖前自动保存一个 `.bak` 备份文件。
2. 在确认窗口中展示旧内容摘要。
3. 加一个“打开生成结果”的按钮。
