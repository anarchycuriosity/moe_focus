# 13. 撤销历史 Release 并合并分支

## 问题背景

当前仓库已有功能整体进入可用状态，但历史 release 存在问题，继续保留会误导后续安装包、tag 和发布说明的判断。因此本次不继续修复 release 包，而是先撤销全部历史 release，再把各分支已经完成的功能合并回主分支。

## 处理思路

1. 先检查工作区状态，确认当前主分支上已有未提交变更。
2. 保留现有变更，避免为了切换分支而丢失工作区内容。
3. 查询 GitHub release 与 tag 状态，确认需要撤销的发布记录。
4. 删除 GitHub release 记录，并删除对应远端 tag 与本地 tag。
5. 将远端功能分支逐个合并到 `main`。
6. 运行项目验证命令，最后推送 `main`。

## 风险点

- `v1.0.0` 在 GitHub release 列表中同时存在正式记录和草稿记录，需要按 release ID 删除，不能只按 tag 删除一次。
- 当前工作区存在大量未提交改动，直接切换分支或重置会造成丢失风险，因此必须先提交。
- `codex/release-rewrite-lab` 分支在另一个 worktree 中被检出，不能在当前 worktree 直接切换该本地分支，但可以合并它的远端跟踪分支。

## 结果记录

已确认 GitHub release 记录包含两个 `v1.0.0` 记录和一个 `v1.1.0` 记录，因此按 release ID 逐个删除。随后删除远端与本地的 `v1.0.0`、`v1.1.0`、`mobile-v1.1.0` tag。

分支检查结果显示，当前 `main` 已经包含所有本地分支与远端分支的提交，没有发现未合入的分支头。因此最终只需要把当前主分支提交推送到 `origin/main`，让远端主分支同步到最新状态。
